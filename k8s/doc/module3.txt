Your facts are mostly correct, but the comment block can be tightened to be precise about what Helm actually does during upgrade --install, and to avoid a small ambiguity:


Helm releases are namespaced.
--namespace tells Helm where to look for the release record and where to create/update the release record.


Helm will not “move” a release to another namespace.
If you run the same release name with a different --namespace, Helm treats it as a different release scope.


upgrade --install behavior is:

If a release named release1 exists in the specified namespace, Helm performs an upgrade there.
If it does not exist in that namespace, Helm performs an install there (creating a new release record), even if a release named release1 exists in some other namespace.



Important subtlety about resources:
Even if you install/upgrade in a namespace, a template can still force metadata.namespace to something else. Best practice is to avoid hardcoding metadata.namespace and rely on .Release.Namespace. But Helm will apply whatever the manifest says.
